Support Bundles give your customers a structured way to share diagnostic data when they report an issue. A customer creates a bundle, runs a single `curl` command on their deployment host, and the collected data appears in your Vendor Portal. Nothing is uploaded without the customer's explicit approval.

<Aside type="note">
  Support Bundles are available on Trial, Pro, and Enterprise plans. The feature
  must be enabled per customer organization before customers can use it. See
  [Enabling features per
  customer](/docs/guides/customer-management/#enabling-and-disabling-features-per-customer).
</Aside>

## How it works

1. The customer creates a support bundle from the Customer Portal, giving it a title and optional description.
2. Distr generates a one-time `curl` command tied to that bundle. The command is valid for 24 hours.
3. The customer runs the command on the affected host. An interactive script launches that collects the environment variables you have configured and the logs of each Docker container. The customer selects which containers to include before anything is sent.
4. Once the script finishes, the bundle appears in the Vendor Portal with all collected resources attached.
5. You and the customer can exchange comments on the bundle. When the issue is resolved, you mark it as resolved.

## Vendor setup

### Configure environment variables

Before customers can create bundles, you need to configure which environment variables should be collected. The script will only collect variables on this list. Variables marked as **Redacted** have their values replaced with `[REDACTED]` in the uploaded data.

In the Vendor Portal sidebar, navigate to **Support Bundles**, then click **Settings**.

<ThemedImage
  lightSrc={settingsLight}
  darkSrc={settingsDark}
  alt="Support Bundle Settings showing the environment variable configuration"
/>

To add variables manually, click **Add Variable** and enter the variable name. To import from an existing file, click **Import** and paste the contents of a `.env` or config file. Lines in the format `KEY=value` or `KEY: value` are accepted; only the variable name is imported, not the value. Lines starting with `#` are ignored.

Toggle **Redacted** for any variable whose value should not appear in collected bundles.

Click **Save** when done.

### Enable the feature for a customer

Support Bundles must be enabled per customer organization before customers can create bundles. See [Enabling features per customer](/docs/guides/customer-management/#enabling-and-disabling-features-per-customer) for how to do this.

## Reviewing support bundles

In the Vendor Portal, navigate to **Support Bundles** to see all bundles across all customers.

<ThemedImage
  lightSrc={listVendorLight}
  darkSrc={listVendorDark}
  alt="Support Bundles list in the Vendor Portal"
/>

The list shows each bundle's title, customer, status, resource count, comment count, and when it was last updated. Click any bundle to open the detail view.

In the detail view:

- **Collected resources** are listed under **Resources**. Click any resource name to expand and read its content.
- **Comments** appear below the resources. Both vendor and customer users can leave comments.
- If the bundle is still in the **Initialized** state (the customer has not yet run the collection script), the collect command is shown so you can share it again if needed.
- Click **Mark as Resolved** to close the bundle once the issue is addressed.

## Customer flow

When Support Bundles is enabled for a customer organization, customers see **Support Bundles** in their Customer Portal sidebar.

To create a bundle:

1. Navigate to **Support Bundles** and click **New Support Bundle**.
2. Enter a **Title** (required) and an optional **Description** of the issue.
3. Click **Create**. Distr generates a `curl` command valid for 24 hours.
4. Run the command on the affected host. The script will:
   - List available Docker containers and prompt for which ones to include
   - Collect the configured environment variables from the host
   - Collect logs from the selected containers
   - Upload everything to Distr
5. No data is sent until the final confirmation step in the script.

<Aside type="tip">
  The collect command is also visible in the bundle detail view in the Customer
  Portal for as long as the bundle is in the **Initialized** state, so customers
  can retrieve it again if needed.
</Aside>

## Bundle status lifecycle

| Status          | Meaning                                                                         |
| --------------- | ------------------------------------------------------------------------------- |
| **Initialized** | Bundle created, collection script not yet run. Secret valid for 24 hours.       |
| **Created**     | Collection script ran successfully. Resources are attached and ready to review. |
| **Resolved**    | Vendor has marked the bundle as resolved.                                       |
| **Canceled**    | Bundle was canceled before data was collected.                                  |

Either party can cancel an **Initialized** bundle. Only vendors can resolve a bundle. Once resolved or canceled, a bundle cannot be reopened.
